# 快速开始

## 1. 先从入口开始

用自然语言提出一个业务问题：

```text
[$hs-entry] 我想知道某个 KPI 为什么下滑，以及需要哪些证据来判断。
```

`hs-entry` 会先读取业务图谱、指标索引和 Source 卡，再判断应该走 onboarding、指标树维护、内部分析、外部研究，还是输出格式整理。标准和重型任务会先给出施工图，等待你确认。

## 2. 建立第一版业务图谱

如果这个业务还没有被结构化，使用：

```text
[$hs-onboarding] 帮我为这个业务建立第一版业务认知地图。
```

这一步不追求完美，目标是先得到一版可用的业务图谱：业务框架、指标资产、维度、数据源、业务事实和待补材料清单。

## 3. 维护业务图谱

当你发现缺指标、维度错、数据源说明过期、链接失效时，使用：

```text
[$hs-graph] 把这个指标/数据源/业务事实补充到当前业务图谱里。
```

`hs-graph` 应该同时更新资产文件、索引、manifest 和地图相关绑定关系。

## 4. 做分析

当业务图谱已经存在，并且问题已经明确时：

```text
[$hs-analysis] 这个指标为什么发生变化？
```

如果涉及外部样本、竞对、公开案例或市场机会：

```text
[$hs-research] 对标这个外部机会，并映射回我的业务图谱。
```

当内部分析需要读取、计算或生成复杂表格时，标准/重型任务会在施工图确认后依次建立数据契约、完成中间测算、通过指标审计，再构建最终表格与报告。用户仍然只需要从 `hs-entry` 发起问题。

## 5. 整理输出

如果需要报告、HTML、Excel、图表或可读性更强的摘要，通过相关流程调用 `hs-output`。

## 6. 反馈与回写

如果结果有错误、不完整、难读，或者需要写回系统：

```text
[$hs-feedback] 这次结果漏了一个重要指标，请帮我判断问题类型，以及应该回写到哪里。
```

Feedback 是系统自我改进回路。它会判断这个问题应该修业务图谱、修 Skill 规则、修输出契约，还是只记录为一次执行提醒。
