# 全渠道电商零售 Demo：演示命题与标准答案

> 这是一份合成业务的测试设计，不是对任何真实企业的判断。数据生成必须服务于这些预先确认的业务事实，而不是事后从随机数里拼故事。

## 2. Case 01：某月收入为什么下滑

### 用户命题

分析 2026 年 5 月净销售额为什么下降，给出主因、次因和抵消因素。

### 应命中的证据路径

```text
净销售额
-> GMV、退款金额
-> 支付订单数、客单价
-> 支付客户数、购买频次
-> 用户漏斗与商品有货率
-> 区域、品类、渠道拆分
-> FACT-0007、FACT-0008
```

### 预期结论

- 主因：东区小家电供给受限，有货率下降，进一步压低详情加购转化和支付订单数。
- 次因：自营 App 付费流量增加，但用户意向偏低，访问增长没有转成详情浏览和支付客户。
- 抵消因素：南区老客购买频次与部分品类客单价保持增长，减轻整体降幅。
- 不能下的结论：不能只凭流量与销售额同步变化，就断言投放是全部下滑原因。

## 3. Case 02：哪个区域综合表现最好

### 用户命题

比较北区、东区和南区在 2025 年以来的综合经营表现，并解释优势来自哪里。

### 比较规则

综合表现不能由单一指标决定，至少同时观察：

- 净销售额增长与规模。
- 贡献利润和贡献利润率。
- 退款率和准时履约率。
- 新客获取与老客贡献。
- 有货率和关键品类结构。

### 预期结论

- 南区是综合表现最稳健的区域：规模不是最大，但净销售额和贡献利润持续增长，老客贡献与购买频次改善，履约和供给保持稳定。
- 东区绝对规模最大，但在仓切换和供应商事件期间波动明显，不能仅凭规模判定最好。
- 北区新客获客成本较低，但退款率和利润质量弱于南区。
- “最好”必须保留评价标准；更换评价目标后，排名可以变化。

## 4. Case 03：提高下一季度目标要承担什么压力

### 用户命题

在自然经营预估基础上提高净销售额目标，判断需要哪些过程指标承担压力，并比较不同实现路径。

### 固定边界

- 第一版不假设新增城市和仓库。
- 在售 SKU 数只允许小幅变化。
- 付费营销费用存在预算上限。
- 退款率和准时履约率是质量约束，不能为了规模无限恶化。

### 应比较的路径

1. 流量驱动：增加付费访问，检查获客成本、付费投放产出比和贡献利润。
2. 转化驱动：改善访问详情、详情加购和提交支付转化率。
3. 客单驱动：提高平均销售单价或单订单件数，检查毛利率和退款率。
4. 复购驱动：提高老支付客户数和购买频次。
5. 平衡路径：由多项可实现的小幅改善共同承担目标差额。

### 预期结论

- 不应让单一效率指标承担全部增长要求。
- 推荐结果应来自多路径情境测算，而不是直接把历史同比套到全部过程指标。
- 当目标超过供给、流量和质量约束时，应输出缺口与资源要求，而不是强行给出可实现结论。

## 5. Case 04：新增商品品类如何更新图谱

### 用户命题

在示例业务中新增“宠物生活”一级品类，判断业务图谱和数据源需要如何更新。

### 预期操作

- 更新品类维度的枚举值。
- 在品类层级中补充一级品类、二级类目和 SKU 的映射。
- 检查核心指标是否均可按新类目拆分。
- 检查订单、库存、财务和履约 Source 是否已有相应字段和值。
- 新建“新品类试运营”业务事实，写清时间、城市和适用范围。
- 更新资产索引、绑定关系和业务认知地图。

### 不能做的事

- 不能因为维度里增加了名称，就默认数据源已有完整数据。
- 不能把试运营局部结果直接推广为全业务规律。

## 6. Case 05：数据源错误如何阻断并回写

### 故意制造的问题

在专用坏样本中删除订单数据的 `refund_amount` 字段，或让 Source 卡指向错误文件。

### 预期行为

1. `hs-entry` 的 Graph Scan 披露 Source 缺口。
2. `hs-data-contract` 将净销售额、退款率和利润相关计算标记为 `blocked`。
3. 不使用空值、0 或兜底公式替代缺失字段。
4. `hs-feedback` 判断是 Source 或数据契约问题。
5. `hs-graph` 修复 Source 卡或明确降低本轮交付范围。

### 验收失败

以下任一行为都视为失败：

- 继续输出完整净销售额归因。
- 把缺失退款金额当作 0。
- 只提示“数据可能不准”，但仍给出确定结论。

## 7. 回归测试意义

标准答案不是要求 AI 逐字复述，而是检查它是否：

- 读全指标链和适用范围。
- 命中关键业务事实。
- 区分主因、次因、抵消因素和口径变化。
- 在证据不足时降低结论强度。
- 遇到坏数据时停止，而不是编造。
