# Business Graph File Contract

这份文件约束 `business_graphs/{business_id}/` 下各类资产文件应该写什么、不能写什么，以及必须如何互相链接。

业务认知地图如何从这些资产中抽取展示字段、统计资产数量和生成跳转清单，见 `map-data-model.md`。本文件是资产写作规范；`map-data-model.md` 是展示模型。两者发生字段定义冲突时，以本文件为准。

## Common Rules

每个资产文件都必须有 YAML frontmatter：

```yaml
type:
id:
name:
status:
updated_at:
```

通用要求：

- `id` 必须稳定，不随文件名或任务变化。
- 文件标题必须包含 `id` 和 `name`。
- 所有跨文件关系必须使用 Markdown 链接，不只写纯文本名称。
- 能链接到已有资产时必须链接；没有资产可链接时，写入“待补”并说明原因。
- 不要在多个文件里重复维护同一段核心定义。一个概念只在它自己的文件中完整定义，其他文件只链接和说明关系。

## metrics/

指标文件回答：这个指标在业务上是什么意思，怎么拆，适用于哪里。

必填要素：

- 节点定位：这个指标是结果、过程、资源、效率、校准或诊断用途中的哪一类，不需要硬贴唯一标签，但要说明用途。
- 定义或计算公式：能公式化就写公式；不能公式化就写清业务定义。
- 可拆解为：结构拆解、公式拆解、条件性拆解或待补拆解。
- 适用范围：适用哪些业务、组织、时间或场景；不适用哪里。
- 拆分维度：链接到相关 `dimensions/` 或 `hierarchies/`。
- 数据源：链接到能取数的 `sources/SRC-*.md`。
- 业务事实：链接到会影响使用的 `facts/FACT-*.md`。
- 关系备注：说明容易误用的关系，例如条件成立范围、不能直接相加、不能泛化。

边界：

- 指标文件不负责解释真实数据文件长什么样。
- 指标文件可以链接 source，但不要复制完整 source 字段说明。

### 拆解类型定义

指标文件中的 `可拆解为` 必须区分以下四类关系。

#### 结构拆解

定义：一个结果指标可以按业务构成被拆成若干组成部分，组成部分通常可以汇总回主题指标。

典型问题：这个结果由哪几块业务构成？

例子：

```text
业绩 = 新签业绩 + 续费业绩 + 充值业绩 + 框架业绩
框架业绩 = 新框业绩 + 续框业绩
```

边界：

- 结构拆解强调“业务组成”，不是计算机制。
- 结构项是否能直接相加，要看口径是否互斥、完整、同单位；不满足时必须写明限制。

#### 公式拆解

定义：一个指标可以通过明确数学公式被解释或计算，公式两侧存在稳定计算关系。

典型问题：这个指标可以由哪些分子、分母、效率或单价算出来？

例子：

```text
业绩 = 签约客户数 × 签约客户ARPU
新签业绩 = 新签签约客户数 × 新签ARPU
总人效 = 业绩 / 月均在职人数
到期续费率 = 到期续费客户数 / 到期客户数
```

边界：

- 公式拆解可以进入计算和反推。
- 率类、人效、ARPU、亩产等派生指标汇总时必须用基础分子分母重算，不能直接相加或求平均后冒充总值。

#### 条件性拆解

定义：一个拆解关系只在特定业务、组织、维度、时间或场景下成立，不能泛化为全局关系。

典型问题：这个解释路径在什么条件下才成立？

例子：

```text
新签业绩 = 商机量 × 亩产
仅限：电销新签团队的新签业绩解释。

关键期续费率
仅限：四大区续费业务。
```

边界：

- 条件性拆解必须写清适用范围和不适用范围。
- 如果条件缺失，分析时只能标为 `待判断` 或 `待补数据`，不能默认套用。
- 条件性拆解常常需要链接 `facts/` 或 `hierarchies/` 来说明为什么成立。

#### 待补拆解

定义：当前知道指标还可以继续解释，但缺少业务机制、数据源、字段或口径，暂时不能写成结构、公式或条件性拆解。

典型问题：这个指标为什么变化，理论上还应该往哪里追，但现在缺什么？

例子：

```text
非电销新签团队的新签业绩
待补：真实新签来源、客户来源、销售动作或流量来源。

商户复购下降
待补：商户流失原因、缺货率、价格竞争力、履约失败明细。
```

边界：

- 待补拆解不是结论，而是缺口清单。
- 后续补齐业务事实或数据源后，必须改写为结构拆解、公式拆解或条件性拆解。

## dimensions/

维度文件回答：这个单一切片口径是什么。

适用场景：

- 月份、城市、客户类型、产品形态、业务动作、客户状态等单个维度。
- 即便这个维度参与某个层级关系，它自己的定义仍放在 `dimensions/`。

必填要素：

- 维度定义：这个维度如何理解。
- 可取值或取值来源：列出主要枚举值，或说明从哪个 source 字段获得。
- 适用指标：链接到可按该维度拆分的 `metrics/IND-*.md`。
- 数据源字段：链接到 `sources/SRC-*.md`，并说明对应字段名。
- 相关 hierarchy：如果它参与多层映射，链接到对应 `hierarchies/HIER-*.md`。
- 注意事项：空值、历史变更、口径例外或不可直接比较的情况。

边界：

- `dimensions/` 不维护多维度完整映射表。
- 如果一个维度必须依赖上游或下游维度才能被正确理解，应在 `hierarchies/` 建中控文件。

## hierarchies/

层级文件回答：多个维度之间如何稳定嵌套、映射或互相约束。

适用场景：

- 组织架构存在一级、二级、三级、城市、团队、dept_type 等多层嵌套。
- 同一个子项在不同口径下有特殊归属。
- 多个维度构成网状映射，仅靠单个 dimension 文件会让含义变模糊。

必填要素：

- 层级目的：为什么需要这个 hierarchy，而不是普通 dimension。
- 参与维度：链接到所有相关 `dimensions/DIM-*.md`。
- 映射关系：用表格列出关键映射，或指向权威 source 字段。
- 适用范围：适用哪些指标、业务线、时间段或分析场景。
- 例外规则：特殊组织、历史调整、合并计算、缺失值处理。
- 支撑数据源：链接到能验证映射关系的 `sources/SRC-*.md`。
- 影响指标：链接到会被该层级影响的 `metrics/IND-*.md`。

边界：

- 不要为普通枚举值创建 hierarchy。
- hierarchy 是多维度关系的中控，不是维度定义的替代品。

## facts/

业务事实文件回答：有哪些会影响指标使用和分析判断的事实。

必填要素：

- 事实陈述：一句话写清事实。
- 适用范围：业务、组织、时间或场景边界。
- 影响对象：链接到相关 `metrics/`、`dimensions/`、`hierarchies/` 或 `sources/`。
- 证据来源：用户说明、历史报告、数据源、会议结论或待验证来源。
- 分析影响：这个事实会如何改变计算、排除、解释或展示。

边界：

- fact 不是观点池。未经验证的判断要标注为假设或待验证。
- 不要把完整指标定义写进 fact。

## sources/

数据源文件回答：顺着指标树判断出要看某个指标后，AI 去哪里取数。

必填要素：

- 文件、SQL、表格或系统位置。
- 数据粒度：一行代表什么。
- 关键字段：字段名、含义、可支持的指标或维度。
- 支撑指标：链接到 `metrics/IND-*.md`。
- 支撑维度：链接到 `dimensions/` 或 `hierarchies/`。
- 数据截止时间、刷新频率或版本说明。
- 权威范围：哪些指标以它为准，哪些不能以它为准。
- 已知限制：缺失、历史口径变更、字段冲突、异常值或不适用范围。

边界：

- source 不重新解释业务逻辑。
- source 只登记真实存在或用户明确提供的数据源。未提供的数据进入待补清单。

## indexes/

索引文件回答：当前业务树里有哪些资产。

必填要素：

- `ASSET_INDEX.md`：列出所有指标、维度、层级、事实、source、业务节点和地图。
- `METRIC_INDEX.md`：列出所有指标的 id、名称、文件路径和一句话用途。

边界：

- 索引不承载完整定义。
- 分析时不能只读索引就下结论，必须回到原始资产文件。
