# Three-Layer Blueprint

三层表格蓝图是从指标树到交付物之间的业务边界编译结果。

## Layer 1: 数据源层

目的：承接原始数据，不做业务判断。

必须记录：

- 数据源名称
- 数据截止时间
- 字段含义
- 数据粒度
- 是否缺字段
- 是否缺月份
- 0 和空白的含义
- 层级维度空值的含义，以及是否需要填为“未展开”或“无下级层级”
- 是否存在合并单元格、跨文件链接、不可机读区域

输出：

- 数据源清单
- 字段口径表
- 数据可用性判断
- 待补数据清单

## Layer 2: 中间测算层

目的：按测算机制组织计算，而不是按最终汇报格式组织计算。

中间测算层可以有多张表，数量由复杂度决定。

常见表：

- 基准情境测算表
- 目标或场景压力反推表
- 漏斗转化测算表
- 供给效率测算表
- 资源效率测算表
- 维度分摊表
- 口径校准表

拆表规则：

- 主题指标不同，倾向拆表。
- 测算机制不同，必须拆表。
- 计算粒度不同，必须拆表。
- 预测和展示不同，必须拆表。
- 内部计算和外部汇报不同，倾向拆表。

标准诊断的最低要求：

- 必须先列出中间表清单，再给结论。
- 每张中间表必须说明它解释哪个主题指标、沿用哪条指标拆解链、使用哪些过程指标。
- 如果过程指标缺少数据，也要保留这张中间表，并把输出字段标为“待补数据”或“本轮不可验证”。
- 不允许只用一张组织贡献表替代完整中间层。
- 如果行维度包含多层组织、地区、品类或客户层级，空层级必须显式保留为“未展开”或等价占位值，不能在透视、分组或 join 时被默认丢弃。

中间表结构模板：

```markdown
### 中间表：<表名>
- 主题指标：
- 解释对象：
- 指标链：
- 行维度：
- 列指标：
- 输入数据源：
- 输出字段：
- 校验方式：
- 本轮状态：已计算 / 待补数据 / 不适用
```

## Layer 3: 最终呈现层

目的：服务交付对象，而不是承载全部计算。

最终呈现层应该包含：

- 核心结论
- 关键指标
- 关键过程指标压力
- 可调输入区
- 风险提示
- 数据截止时间和口径说明

最终呈现层不应该：

- 承担复杂公式
- 隐藏关键假设
- 混入大量内部校验字段
- 让读者无法判断哪些数字可修改，哪些数字是计算结果
