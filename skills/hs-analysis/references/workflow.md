# Hs Analysis Workflow

这份文件细化 `SKILL.md` 中的执行流程。它只解释“用户提出商业分析命题后，应该如何推进”，不引入额外模块名。

## 1. 明确命题

先判断用户真正要完成的动作。

常见命题：

- 经营诊断：解释指标为什么变化。
- 亮点证明：找到支持某人、某团队或某项目值得被认可的事实。
- 复盘 / 述职：把历史工作转换成可被组织认可的贡献。
- 目标测算：测算未来目标和过程指标压力。
- 方案论证：证明一个方案值得资源投入。
- 行业 / 竞对 / 财报研究：把外部事实转成对自身业务有用的判断。

命题定义必须包含：

- 交付对象：给自己看、给老板看、给业务负责人看、给客户看，还是用于公开表达。
- 决策动作：要定目标、争资源、解释异常、证明成绩、汇报进展，还是判断机会。
- 时间窗口：本月、QTD、季度、年度、活动周期，或指定日期段。
- 对比口径：同比、环比、目标、竞对、大盘、相似区域、历史峰值。
- 交付强度：轻量、标准、重型。

交付强度：

- 轻量：可以只做口头判断，但要说明指标展开路径和下一步需要补的表。
- 标准：默认强度。必须输出证据结构、中间表设计、关键计算和校验结果。
- 重型：需要真实生成 Excel、报告、看板或可交付文件；必须实际建表、写公式、校验并保存。

## 2. 读取指标树

读取指标树的目标，是找到能回答命题的业务逻辑路径，而不是机械给指标贴标签。

先读取指标节点索引。索引只负责列出全量节点，不负责解释业务关系。业务关系、公式、适用范围、数据源和事实，仍以各节点 MD 原文为准。

必须读取：

- 指标节点索引。
- 主题指标。
- 主题指标的结构拆解和公式拆解。
- 适用范围与不适用范围。
- 维度层级和 HIER 关系。
- 业务事实。
- 数据源口径和数据源缺口。

### 2.0 临时指标审计表

标准和重型任务必须先输出临时指标审计表，再进入中间表设计和计算。轻量任务可以压缩展示，但必须说明主题指标、主要纳入指标和被跳过指标的原因。

临时指标审计表的目的，是让用户能检查“有哪些指标被看见、哪些指标被排除、为什么排除”。它不是最终结论，也不是指标树真源。

表格字段：

| id | name | type | 本轮状态 | 判断理由 | 是否读取原文 | 后续动作 |
|---|---|---|---|---|---|---|

状态只能使用：

- `纳入`：本轮必须进入证据结构或计算。
- `排除`：本轮确认不需要，但必须写原因。
- `待判断`：可能相关，需要先读原文或补业务边界后决定。
- `不适用`：指标适用范围与本轮对象不匹配。
- `待补数据`：逻辑相关，但缺少数据源、字段或口径。

执行规则：

1. 先从索引中列出主题指标及可能相关的指标、维度、数据源、事实节点。
2. `纳入`、`待判断`、`待补数据` 的节点必须读取原始 MD。
3. 读取原始 MD 后，如果发现新的拆解关系或条件路径，要把新增节点补回临时指标审计表。
4. 不允许因为“看起来不是重点”而沉默跳过一级结构指标。
5. 用户指出漏项时，必须先修正临时指标审计表，再修正计算或结论。

### 2.1 解释闭包

不能只读主题指标的一级拆解。必须继续沿着结构拆解、公式关系、业务机制、适用范围和数据源映射展开，直到形成“解释闭包”。

解释闭包至少回答：

- 主题指标发生了什么变化？
- 变化主要发生在哪些结构里？
- 这些结构的变化还能被哪些过程、资源、效率或校准信息解释？
- 哪些指标因为业务边界不适用，需要排除？
- 哪些指标因为缺少数据源或口径，需要列为缺口？

解释闭包必须能回到临时指标审计表：闭包中出现的每个节点，都要在审计表中有状态和原因。

停止展开的条件：

1. 已到指标树叶子节点。
2. 后续指标不适用于当前业务边界。
3. 后续指标缺少数据源或口径，需要列为缺口。
4. 用户明确要求只做轻量表层判断。

### 2.2 异动诊断的最低展开

当用户问“为什么下降”“为什么变了”“原因是什么”时，最低不能停在组织或业务类型贡献表。

必须至少形成：

- 结果层：主题指标总变化。
- 结构层：按组织、产品、业务类型或关键维度定位变化发生在哪里。
- 解释层：对主要变化项继续追到可解释的过程、资源、效率或校准信息。
- 缺口层：如果解释层没有数据，列出应建的中间表和待补数据。

如果主题指标是业绩类指标，不能默认只看总业绩、新签、人效。必须在临时指标审计表中显式处理续费、充值、框架、客户数、ARPU、人效等结构路径；不纳入时写清原因。

## 3. 建立证据结构

先判断本次任务属于哪一种分析方式。

### 3.1 描述性统计

描述性统计不利用指标之间的相关关系产生新计算，主要回答：

- 发生了什么？
- 变化在哪里？
- 哪些结构贡献最大？
- 哪些对象表现最好或最差？
- 哪些异常值得继续查？

常见动作：

- 汇总。
- 排序。
- 对比。
- 拆分。
- 贡献分析。
- 异常定位。

### 3.2 关系型测算

关系型测算利用公式关系、业务机制或明确假设产生新计算，主要回答：

- 未来可能是多少？
- 目标压力落在哪？
- 哪个变量变化会带来最大影响？
- 要达成目标，需要哪些过程指标变化？
- 不同情景下结果会如何变化？

常见动作：

- 目标测算。
- 压力反推。
- 情景模拟。
- 公式归因。
- 变量敏感性分析。

### 3.3 三层表格

三层表格把“原始数据、计算过程、交付表达”分开。

- 数据源层：保留原始数据和原始口径，不做业务判断。
- 中间测算层：承接计算、拆解、归因、预测和校验，让公式和逻辑可追溯。
- 最终呈现层：只放交付对象需要看的结论和关键证据，不承担复杂计算。

标准任务至少要说明：如果继续执行，需要哪些数据源表、哪些中间表、最终呈现哪些字段。

中间表清单必须覆盖临时指标审计表中所有 `纳入` 和 `待补数据` 的关键路径；如果不覆盖，必须写明原因。

### 3.4 从断点到根因

当任务询问“为什么”时，必须区分三个证据层级：

1. **损失来源**：通过贡献拆解确认变化主要发生在哪个组织、产品、渠道、客户群或其他结构。
2. **过程断点**：通过漏斗、资源或效率指标确认变化在哪个业务环节发生。
3. **根因机制**：解释是什么因素导致了这个断点。

前两层不能自动推出第三层。业务事实与指标异常在时间、范围或方向上相符，只能提高候选假设的优先级，不能单独证明因果。

如果根因证据不足，必须输出根因假设矩阵：

| 候选根因 | 现有支持 | 反证或判断边界 | 待补数据 | 当前状态 |
|---|---|---|---|---|

状态至少区分：`已确认`、`优先假设`、`待排查`、`待补数据`、`已排除`。只有存在直接机制指标、稳定对照、反事实、实验或其他可验证证据时，才能把候选假设升级为已确认根因。

## 4. 建表与计算

中间表要围绕稳定计算单元建立。

稳定计算单元由以下条件共同定义：

- 同一主题指标。
- 同一测算机制。
- 同一计算粒度。
- 同一数据来源组合。
- 同一校验方式。

标准诊断的中间表最低字段：

- 表名。
- 主题指标。
- 解释对象。
- 行维度。
- 列指标。
- 输入数据源。
- 输出字段。
- 可验证问题。
- 校验方式。

如果只做口头交付，也要把这张表的结构讲清楚。否则结论只能标为“初步观察”，不能标为“诊断结论”。

### 4.1 四类建表错误

建表和计算时，优先防止四类错误。

第一，把派生指标直接相加。

率类、人效、ARPU、亩产、转化率等指标，不能从明细行直接求和。汇总层必须用分子分母或基础指标重算。

第二，把展示维度误当成因果变量。

组织、城市、渠道、品类、客户类型等维度可以帮助定位变化发生在哪里，但不能自动解释为什么发生。因果或机制需要额外业务事实、公式关系或验证。

第三，把资源变化和效率变化混成一句空话。

人数、客户池、商机量、供给量等资源变化，和人效、转化率、ARPU、续费率等效率变化，要分开表达。否则会变成“业绩下滑因为业务不好”这种无效解释。

第四，把层级维度空值当作不存在。

多层组织、品类、地区、客户分层等维度中，某些对象可能没有下一级层级，例如有二级组织但没有三级城市群。建中间表或透视表前，必须把这类空层级显式保留为“未展开”“无下级层级”或等价占位值，不能让透视、分组、join 默认丢弃这些行。否则会只保留层级更完整的对象，形成错误的局部结论。

### 4.2 合并同类测算

不要默认按业务线拆表。先看哪些对象共享同一种测算机制。

可以合并处理：

- 公式关系相同。
- 适用范围相同或高度相似。
- 计算粒度相同。
- 数据源相同或可以稳定合并。
- 校验方式相同。

必须拆开处理：

- 适用范围不同。
- 业务机制不同。
- 一个需要预测，一个只需要展示。
- 一个有明确资源项，一个没有资源项。
- 一个是内部销售，一个是外部渠道或大客口径。

### 4.3 计算粒度

先选稳定计算粒度，再决定展示粒度。

原则：

- 低基数明细不直接做预测。
- 多层组织时，优先在稳定上级层测算，再向下分摊。
- 城市、客户、人等细粒度通常更适合作为展示或分摊，不默认作为预测粒度。
- 基准期和目标期必须使用同一套层级逻辑，不能一个从明细汇总，一个从上层预测。

## 5. 校验并交付

### 5.1 校验

必须检查：

- 汇总是否等于明细。
- 公式关系是否成立。
- 派生指标是否由分子分母重算。
- 比率变化是否使用 PP。
- 0 与空白是否混淆。
- 维度层级空值是否被显式保留，是否在透视、分组或 join 时被误删。
- 数据源截止时间是否一致。
- 指标适用范围是否被误用。
- 是否有外部链接、跨表错位、公式损坏、循环引用。
- 指标展开是否形成解释闭包，是否存在只读一级拆解就停止的问题。
- 临时指标审计表是否输出，且每个被排除或不适用的指标都有原因。
- 中间表是否覆盖了主要变化项的解释指标。
- 是否把损失来源或过程断点误写成根因；业务事实相符但缺少直接机制证据时，是否已降级为候选假设并列出替代解释和待补数据。

### 5.2 呈现

最终呈现层服务交付对象，不服务计算过程。

原则：

- 只保留能支撑命题的核心字段。
- 保留必要的可调输入。
- 保留关键过程指标压力。
- 不把中间测算细节全部堆给读者。
- 必须明确单位、对比口径、数据截止时间和风险。

### 5.3 回写

执行中发现的问题必须变成指标树回写项。

常见回写：

- 新增业务事实。
- 修正指标适用范围。
- 补充维度 HIER。
- 回填执行中暴露的建表和校验规则，例如层级空值保留、稳定粒度选择、口径错位处理。
- 标注数据源缺口。
- 删除错误指标并同步上下游关系。
- 标注经验系数来源和适用条件。
