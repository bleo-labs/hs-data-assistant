# Regression Workflow

回归样本用于证明：一个已经确认的系统缺陷被修复后，同类任务不会再次以同样方式失败。

它不是案例收藏，也不是把所有用户纠正都升级成产品规则。

## 1. 什么时候建立

满足以下任一条件，且用户已确认修改方向时，可以建立回归样本：

- 同类失败已经重复出现。
- 缺陷会让未来任务产生高风险错误。
- 用户明确确认这是一条可迁移的通用规则。
- 规则修改涉及入口硬约束、数据边界、公式校验、范围闸门或发布门槛。

以下情况只记录，不建立通用回归样本：

- 缺失的是某个私有业务事实或数据源。
- 用户临时改变偏好或任务范围。
- 现有规则已经明确覆盖，只是本轮偶发没有执行。
- 修复方式只适用于一个具体业务。

偶发未执行可以建立“执行提醒记录”，但不能据此修改通用 Skill。

## 2. 样本最小结构

使用 `templates/feedback/regression_case.md`，至少写明：

- 原始命题或最小复现输入。
- 必要前置材料。
- 失败表现。
- 根因层级和被修改的真源。
- 预期行为。
- 禁止行为。
- 验证步骤与通过标准。
- 隐私状态。

## 3. 验证方法

1. 保留原 Case；涉及私有数据时，制作最小匿名版本。
2. 完成经用户确认的修改。
3. 从最早受影响节点重新运行，不只重写最终答案。
4. 对照“预期行为、禁止行为、通过标准”检查。
5. 将结果写回 `run_record`：`pass`、`fail` 或 `conditional`。
6. `fail` 时不得宣称问题已解决；回到根因定位。

## 4. 保存边界

- 私有 Case：保存在本次任务目录的 `regression/`，不进入公开仓库。
- 可公开的匿名通用 Case：经用户确认后，才可进入产品包的测试或示例资产。
- 任何回归样本都不得包含公司名、内部指标值、私有路径、凭证或可反推业务身份的信息。

第一版不要求自动执行测试。先保证样本可读、可复现、可判断通过与否；只有样本积累到足以证明重复价值时，再升级自动化测试。
