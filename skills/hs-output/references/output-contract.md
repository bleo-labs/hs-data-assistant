# Hs Output Contract

这份文件定义 Hs 常见产出物的最低结构。它不是固定模板，而是防止产出物不可读、不可复查、不可交付。

## 1. 任务启动卡

用于 `hs-entry` 在正式执行前对齐任务。

必须包含：

- 用户表面问题：
- 我理解的真实问题：
- 目标交付物：
- 交付对象：
- 时间窗口：
- 证据来源：
- 推荐路径：
- 任务强度：轻量 / 标准 / 重型
- 执行前缺口：
- 需要用户确认的决策：

写法要求：

- 不超过一屏。
- 不提前给复杂结论。
- 对标准和重型任务，必须给出用户可读施工图并等待确认。

## 1.1 施工图 Markdown

用于 `hs-entry` 在标准或重型任务正式执行前，让用户确认执行路径。

施工图 Markdown 是给用户看的执行路线。它优先使用 Mermaid 线框路线图展示节点关系，再用必要字段和表格承载信息。

必须包含：

- 这次要解决什么：真实问题、交付对象、预期产物。
- 我会先看哪些范围：业务对象、时间窗口、指标或研究路径、数据源或样本来源。
- 执行路线：3-5 个步骤，每一步说明要做什么和调用哪个 Hs 能力。
- 这次先不做什么：明确不覆盖范围和暂不下的结论。
- 需要用户确认：范围、交付物、是否开始第一步。

写法要求：

- 优先使用 Mermaid 线框路线图 + 普通 Markdown 标题、列表和必要表格。
- 线框图只展示节点和依赖关系；范围、证据、产物、边界和确认项必须继续用文字说明。
- 图里的每个主要节点，必须能在后台 `project_graph` 的 `Nodes` 中找到对应项。
- 标准和重型任务输出施工图 Markdown 后必须停住，等用户确认。
- 用户修改范围或交付物时，先更新施工图，再继续。

## 2. 临时指标审计表

用于 `hs-analysis` 展示“哪些指标被看见，为什么纳入或排除”。

字段：

| id | name | type | 本轮状态 | 判断理由 | 是否读取原文 | 后续动作 |
|---|---|---|---|---|---|---|

状态只能使用：

- 纳入
- 排除
- 待判断
- 不适用
- 待补数据

写法要求：

- 每个排除、不适用、待补数据都要写理由。
- 用户指出漏项后，先修审计表，再修结论。
- 审计表不是最终报告，不要塞入详细计算。

## 3. 分析结论卡

用于轻量或标准任务的第一屏。

结构：

```markdown
## 结论

一句话回答用户问题。

## 关键证据

1. 证据 A：数字 + 对比口径 + 含义。
2. 证据 B：数字 + 对比口径 + 含义。
3. 证据 C：数字 + 对比口径 + 含义。

## 判断边界

- 数据截止：
- 口径限制：
- 不能证明：

## 下一步

- 如果要继续查，需要补什么表或做什么验证。
```

写法要求：

- 先给判断，再给理由。
- 数字必须带单位和时间窗口。
- 如果证据不足，结论必须降级为“初步判断”或“待验证假设”。

## 4. 标准分析报告

用于标准或重型任务的可交付文字报告。

结构：

1. 摘要
2. 问题与口径
3. 指标路径或研究框架
4. 关键发现
5. 证据与计算说明
6. 风险、缺口与边界
7. 建议或下一步
8. 附录：指标审计表、中间表清单、来源清单

写法要求：

- 摘要必须可单独阅读。
- 摘要里的每个关键判断，正文中必须有图表或关键表格支撑。
- 主体只放支撑判断的证据，不堆全部中间过程。
- 附录承接复查材料。
- 若报告需要发送或传播，默认从本地 HTML 导出 PDF；不把本地 HTML 当唯一传播物。

## 5. 图表 / HTML 简报

用于增强关键结论表现力。

每张图必须回答一个明确问题：

- 标题：写洞察，不只写指标名。
- 副标题：写时间窗口、口径、单位。
- 图例：只保留必要分类。
- 注释：标出异常点或关键转折。
- 来源：标注数据源和截止时间。

图表简报结构：

1. 标题与一句话摘要
2. 结论总览：3-5 条，每条对应图表或关键表格
3. 关键图表 3-6 张，每张图下面保留一句解释
4. 结论边界
5. 后续要补的数据或验证

更多 HTML、PDF 与图表原则见 `report-and-html.md`。

## 6. 业务认知地图

用于 `hs-onboarding`。

默认先在对话内展示，不强制生成 HTML 页面。

对话内业务认知地图必须包含：

- Mermaid 业务框架图。
- 每个节点的资产数量：指标、维度、数据源、业务事实。
- 节点资产汇总表。
- 本地文件跳转清单。
- 待补材料提示。
- 说明：业务框架可修改，地图由指标资产自动渲染。

HTML 仅作为可选导出。用户明确需要浏览、分享或阶段性整理时，HTML 页面必须包含：

- 业务框架主图。
- 每个节点的资产数量：指标、维度、数据源、业务事实。
- 点击数量后可跳转到对应资产清单。
- 待补材料提示。
- 说明：业务框架可修改，地图由指标资产自动渲染。

不要给节点贴“已掌握 / 可升级 / 缺数据源”这类让用户困惑的状态。优先展示资产数量和清单。

## 7. 待补清单 / 回写卡

用于记录本轮任务发现但尚未处理的缺口。

待补清单字段：

| 类型 | 现象 | 影响 | 建议补充 | 建议位置 | 优先级 |
|---|---|---|---|---|---|

回写卡字段：

```markdown
## 回写卡

- 发现的问题：
- 造成的影响：
- 应回写为：
- 建议位置：
- 是否需要用户确认：
- 本次是否已处理：
```

写法要求：

- 不把临时假设直接写成系统规则。
- 需要用户确认的，不要静默改真源。
