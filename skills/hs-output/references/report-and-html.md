# Report, HTML And PDF Standard

这份文件约束 Hs 的分析报告、HTML 简报和 PDF 分享版。

## 1. 交付形态

默认形态：

- 本地 HTML：用于阅读、检查、演示和迭代。
- 配套 Excel / CSV：用于复查数据、公式和中间表。
- Markdown 摘要：用于对话内快速确认结论。

需要分享时：

- 优先从本地 HTML 导出 PDF。
- PDF 只承载最终结论、关键图表、口径边界和必要附录。
- 不在没有稳定托管方案前，把云端 HTML 链接作为默认交付承诺。

不默认使用：

- 第三方 HTML 托管平台。
- 需要备案、域名或长期维护的对象存储托管。
- 只适合临时预览、但不便于留档和转发的链接。

## 2. 报告层级

标准分析报告至少分三层：

1. **第一屏结论层**：一句话结论 + 3-5 条关键判断。
2. **证据展开层**：每条关键判断对应一张图或一张关键表。
3. **复查附录层**：指标审计、中间表清单、数据源、口径边界。

不要把中间表直接堆进正文。正文服务读者判断，附录服务复查。

## 3. 图表支撑原则

分析总览和关键结论必须有图表或关键表格支撑。

判断分级：

- 有图表或关键表格支撑：可以写成“结论”。
- 只有局部数字支撑：写成“初步判断”。
- 只有业务解释或经验判断：写成“假设”。
- 数据源或口径缺失：写成“待验证问题”。

每张图必须回答一个明确问题：

- 标题写洞察，不只写指标名。
- 副标题写时间窗口、口径、单位。
- 图下保留一句解释：这张图证明什么。
- 异常点、转折点、口径变更必须标注。
- 图表不支持的结论，不要写进同一段。

## 4. 图表选择

按问题选择图表：

- 看趋势：折线图。
- 看结构：堆叠柱状图、结构占比条形图。
- 看贡献：瀑布图、贡献拆解表、横向条形图。
- 看对比：分组柱状图、排名表。
- 看分布：箱线图、散点图。
- 看路径：指标树图、漏斗图。
- 看经营判断：结论卡 + 关键证据图。

如果数据不适合画图，使用关键表格，但表格必须突出结论列、对比列和风险列。

## 5. HTML 报告结构

本地 HTML 推荐结构：

```markdown
1. 标题与一句话结论
2. 关键结论总览
3. 关键图表区
4. 证据与计算说明
5. 判断边界
6. 下一步建议
7. 附录入口
```

HTML 页面要求：

- 第一屏能看到结论。
- 每个图表模块独立成块，标题、图、解释、来源齐全。
- 使用简洁配色，不用装饰性动效。
- 图表优先可截图、可复制到文档。
- 数据截止时间和口径说明必须可见。

## 6. PDF 分享版结构

PDF 用于发送、传播或汇报，不用于承载全部中间过程。

推荐结构：

```markdown
1. 封面：主题、时间、数据截止、作者/项目
2. 摘要：一句话结论 + 3-5 条关键判断
3. 图表正文：每页一个核心判断
4. 判断边界：不能证明什么、哪些口径有限
5. 下一步建议
6. 附录：必要指标审计和来源清单
```

PDF 规则：

- 关键图表必须能单页阅读。
- 不要把宽表硬塞进正文。
- 如果表格过宽，放入附录或 Excel。
- 需要对外传播时，隐藏内部路径、临时文件名和不该公开的数据字段。

## 7. 与其他 Hs Skill 的关系

- `hs-entry`：决定是否需要 HTML、PDF、Excel、图表或只需要结论卡；`project_graph` 的 Deliverables 优先。
- `hs-analysis`：提供指标审计表、中间测算表、关键结论和图表数据。
- `hs-research`：提供外部样本、来源清单和可引用证据。
- `hs-onboarding`：提供业务认知地图和指标资产清单。
- `hs-graph` 或 Feedback：处理报告中发现的口径缺失、指标缺失和 Bad Case 回写。

`hs-output` 只决定“怎么交付”，不重新发明分析结论。
